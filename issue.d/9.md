# Issue #9：max tool-calling rounds reached - 输入 token 超出上下文限制

**来源**：https://github.com/cprobe/catpaw/issues/9  
**提交者**：@zhch578  
**日期**：2026-03-06  
**状态**：已修复

---

## 问题描述

调用本地模型时，输入 token 达到 342K，远超模型上下文限制。

## 环境信息

- 模型：GLM-4.7
- 部署：本地 litellm 代理
- 上下文配置：8192

## 错误信息

```
max tool-calling rounds reached
token: in=342640 out=2758 total=345398
```

## 预期行为

应该自动截断或限制上下文，而不是超出限制。

---

## 分析结论

### 报错来源

`max tool-calling rounds reached` 来自 `chat/session.go` 第 142 行（chat 模式），`token: in=%d out=%d total=%d` 来自 `chat/progress.go` 第 45 行。**本 issue 触发路径是 `catpaw chat` 模式**，不是 diagnose 自动诊断模式。

### 根因

1. **chat 模式 `conversationLoop` 内无上下文管控**  
   `trimHistory` 只在用户发新消息时（`HandleMessage`）调用一次，在 `conversationLoop` 的 20 轮循环内**从不触发**。每轮追加 assistant + tool 消息（单条工具输出最大 32KB），messages 持续增长到数十万 token，全部发给 API。

2. **chat 模式完全没有 `context_window` 感知**  
   `ChatSession` 不读取 `context_window` 配置，不做任何基于 token 的截断。`trimHistory` 按固定 40 条消息截断，对 8K 上下文模型远远不够。

3. **`context_window` 未配置时默认 128000**  
   若用户未显式设置 `context_window`，引擎使用默认 128000（`diagnose/engine.go`），认为有 128K 预算而不做任何限制。

4. **diagnose 模式也有类似问题（潜在）**  
   `estimatedTokens > contextWindowLimit` 时只追加"请输出最终报告"的消息，不对 `messages` 做实际截断。

### 342K token 的构成

342K 是 `totalUsage.PromptTokens` 跨 20 轮的累加值（每轮 prompt 包含之前所有消息）。每轮 prompt 都超过 8K 限制，litellm 可能静默截断输入，导致 AI 看到的上下文不完整，无法给出有意义的回答，只能继续调用工具，恶性循环直到耗尽 20 轮。

## 修复内容

### 1. 新增 `CompactMessages` 函数（`diagnose/aiclient/compact.go`）

chat 和 diagnose 共用的消息压缩函数：
- 始终保留 `messages[0]`（system prompt）
- 将后续消息按"组"划分：独立消息为一组，tool-call 序列（assistant+ToolCalls + 后续 tool 回复）为一组，不可拆分
- 从最老的组开始移除，直到总 token 估算值在预算内
- 配套 `EstimateMessagesTokens` 函数，按消息内容 + 每消息开销 + tool_call 结构开销估算

### 2. chat 模式加入上下文管控（`chat/session.go`）

- `SessionConfig` 和 `ChatSession` 新增 `ContextWindowLimit` 字段
- `conversationLoop` 每轮 `fc.Chat()` 之前调用 `CompactMessages`，确保发送的消息在上下文窗口内
- 保留原有 `trimHistory`（按条数截断）作为第二道防线

### 3. 两个调用方传入 ContextWindowLimit

- `chat/chat.go`（CLI `catpaw chat`）
- `agent/agent.go`（远程 chat session）
- 通过 `ContextWindowLimitFromConfig(cfg)` 计算：`context_window * 80%`

### 4. diagnose 模式加入消息压缩（`diagnose/engine.go`）

- 在 `fc.Chat()` 之前，若 `estimatedTokens > contextWindowLimit`，先调用 `CompactMessages` 压缩
- 压缩后若仍接近上限（90%），再追加"请输出最终报告"提示
- 不再仅靠"提醒 AI 收尾"来防止超限

### 5. 启动时 context_window 下限校验

- `NewDiagnoseEngine` 中检查 `context_window < 16000` 时输出 warn 日志
- 70+ 工具的 system prompt 约 3000-5000 token，8K 上下文基本不可用

### 修改文件清单

| 文件 | 改动 |
|------|------|
| `diagnose/aiclient/compact.go` | 新增：`CompactMessages`、`EstimateMessagesTokens` |
| `diagnose/aiclient/compact_test.go` | 新增：5 个测试用例 |
| `chat/session.go` | `SessionConfig`/`ChatSession` 加 `ContextWindowLimit`；`conversationLoop` 每轮压缩 |
| `chat/chat.go` | 传入 `ContextWindowLimit`；新增 `ContextWindowLimitFromConfig` |
| `agent/agent.go` | 传入 `ContextWindowLimit` |
| `diagnose/engine.go` | `fc.Chat()` 前压缩；启动时 warn 小上下文 |

### 用户侧配置建议

使用小上下文模型时，务必在 `conf.d/config.toml` 中显式设置：

```toml
[ai.models.default]
context_window = 8192   # 必须与模型实际能力一致
```

建议 `context_window` 不低于 16000，否则 system prompt 就可能占满大部分窗口。
