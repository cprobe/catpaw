# catpaw 短帖素材（极简版）

每条控制在 1-2 句话，适合直接发朋友圈 / X / LinkedIn，末尾自行补文章链接即可。

## 1. 告警发出来之后，谁来查根因？

很多告警系统其实只解决了 20% 的问题：它告诉你“坏了”，但不告诉你“为什么坏、先查哪里”。我最近写了一篇关于告警闭环的文章，核心就是把“报警”往前推进到“第一轮根因初筛”。

## 2. Prometheus 很强，但它不负责半夜帮你查机器

Prometheus 很擅长 metrics、趋势和告警规则，但它不直接负责“这台机器现在到底哪里不对、该先查什么”。这篇文章讲的是 catpaw 怎么补上主机侧 check、监控盲区和告警后的第一轮根因分析。

## 3. Redis 出问题时，你到底该先看什么？

排 Redis 故障时，只盯 QPS、CPU、used_memory 往往不够，真正决定定位效率的通常是角色、复制链路、`maxmemory`、`rejected_connections`、cluster fail/pfail 这些更贴近故障的信号。 这篇文章把 Redis 监控拆成“周期检查”和“按需诊断”两层来讲。

## 4. 比故障更可怕的，是系统正在悄悄漂移

很多事故不是突然发生的，而是系统先悄悄偏离了健康基线：`sysctl` 被重置、数据盘没挂上、SELinux 被关掉、NTP 偏移越来越大。 这篇文章讲的是为什么成熟的 SRE 不能只等告警，还要做基线检查和主动巡检。

## 5. 监控你的监控系统：Prometheus 挂了之后，谁来发现？

最危险的时刻，往往不是告警很多，而是告警突然一个都没了，因为这可能意味着监控系统自己已经失声了。 这篇文章讲的是为什么 Prometheus / Nightingale / Alertmanager 也需要一层独立哨兵，以及最小自监控清单该怎么配。
