# 比故障更可怕的，是系统正在悄悄漂移

> **TL;DR**：很多线上事故不是某个进程突然崩了，而是系统在几周、几个月里悄悄偏离了原本的健康状态。内核参数被重置、数据盘没挂上、安全模块被临时关掉后忘记恢复、NTP 漂移越来越大……这些问题在真正炸之前往往没有明显症状。catpaw 值得写的，不只是“出了事怎么查”，还包括另一件更成熟的事：把基线守住，并在变更后主动做一轮巡检。

## 很多故障并不是“突然发生”的

SRE 做久了会有一种很强的体感：

很多事故表面上看像是某个时间点突然爆了，但真正的根因，往往在更早的时候就已经埋下了。

比如：

- 内核升级后，`somaxconn` 悄悄回到了默认值
- 某次重启后，数据盘没有重新挂上，应用还在往根分区写
- 运维临时 `setenforce 0` 排障，事后忘了恢复
- NTP 没彻底挂，但偏移越来越大，日志、证书、认证已经开始不稳定

这些问题有个共同点：

- 它们一开始不一定会触发“服务不可用”
- 大盘通常也不会第一时间变红
- 但它们会让你的系统逐渐偏离安全状态
- 一旦叠加流量、变更或偶发抖动，就会把小偏差放大成事故

所以成熟的运维体系，盯的从来不只是“现在坏没坏”，还包括：

> 这台机器是不是还处在我们认可的健康基线之上？

这就是我觉得很多团队最缺的一层：

**漂移检测。**

## 只做故障告警，通常会让你永远晚半拍

如果你的监控体系只关心下面这些：

- CPU 高不高
- 内存满不满
- 磁盘空间够不够
- 端口能不能通
- 进程还在不在

那它当然有价值，但它只覆盖了“显性问题”。

而生产环境里真正让人头疼的，经常是“系统状态已经偏了，但还没偏到立刻炸”的那一类问题。

这种问题最麻烦的地方，不在于它们难理解，而在于它们很少被系统性地表达成检查项。

比如下面这些场景：

- `sysctl` 参数和你原本调优的值不一致
- 挂载点还在配置里，但运行时已经不存在
- `/tmp`、`/var/tmp`、`/dev/shm` 的安全选项被改掉
- SELinux 从 `enforcing` 变成了 `permissive`
- 时钟已经有明显漂移，但还没到所有请求都失败的地步

这些都不是“看一眼 CPU 曲线”能发现的。

它们需要的不是趋势图，而是**基线比对**。

## 监控系统成熟之后，一定会长出“基线”这层能力

我更喜欢把机器侧监控分成三层：

| 层次 | 你在关心什么 | 常见工具形态 |
| --- | --- | --- |
| 资源与可用性 | 现在是不是已经明显异常 | metrics、存活检测、阈值告警 |
| 故障定位 | 出了问题先查哪里 | check、诊断工具、AI RCA |
| 基线与漂移 | 系统是不是还维持在我们认可的状态 | baseline check、巡检、合规检查 |

很多团队第一层做得还行，第二层开始补，第三层最容易被拖着不做。

原因也很现实：

- 它不会像故障那样立刻打你脸
- 很难在一次事故复盘里被直接归因
- 做起来像“额外工作”

但实际上，第三层恰恰最能决定系统是不是稳。

因为它处理的是故障之前的那个阶段：

> 问题还没形成事故，但系统已经开始偏离。

这也是为什么我觉得 catpaw 里最容易被低估的一组插件，不是 CPU / mem / disk，而是：

- `sysctl`
- `mount`
- `secmod`
- `ntp`

它们都不炫，但都很像真正会在生产环境里救你命的东西。

## 基线检查，首先要盯住“本来应该稳定不变”的东西

有些信号天生适合做趋势分析。

比如 CPU、QPS、延迟。

但还有一类信号，不应该频繁变化。一旦变了，本身就值得警惕。

### 1. 内核参数基线：`sysctl`

`sysctl` 是最典型的漂移源。

因为下面这些事情都会让你原本调好的参数悄悄失效：

- 重启后配置没生效
- 内核升级后默认值变了
- 某个自动化任务覆盖了配置
- 某个模块重新加载

问题是，这类变化通常不会立刻表现成“服务挂了”。

它更常见的表现是：

- 高峰期连接开始排队
- 某些超时慢慢变多
- 某些网络行为和之前不一致

catpaw 的 `sysctl` 插件更像一个“期望值守卫”。

你直接把基线写进去：

```toml
[[instances]]
interval = "60s"

[instances.param_check]
params = [
  { key = "net.core.somaxconn", op = "ge", value = "65535" },
  { key = "vm.swappiness", op = "le", value = "10" },
  { key = "fs.file-max", op = "ge", value = "1000000" },
]
```

然后它持续回答的就是：

> 这些参数现在还是不是你要求的那个状态。

这比“记在 wiki 里，靠人记得去看”可靠得多。

### 2. 挂载与文件系统基线：`mount`

挂载问题也是典型的“平时不看，出事就很痛”的类别。

最常见的几种：

- 数据盘重启后没挂上
- NFS 条目还在配置里，但实际没有按预期挂载
- `/tmp`、`/var/tmp`、`/dev/shm` 的安全选项不对
- 文件系统类型和预期不一致

很多人会把挂载问题和“磁盘监控”混在一起。

但其实它们不是同一层：

- `disk` 更像容量和可写性检查
- `mount` 更像挂载基线检查

比如你可以明确要求：

```toml
[[instances]]
interval = "60s"

[[instances.mounts]]
path = "/data"
fstype = "ext4"
severity = "Critical"

[[instances.mounts]]
path = "/tmp"
options = ["noexec", "nosuid", "nodev"]
severity = "Warning"

[instances.fstab]
enabled = true
severity = "Critical"
```

这个思路很适合两类场景：

- 业务数据盘、备份盘、日志盘的存在性确认
- CIS / 安全基线里对挂载选项的要求

这里有个很实用的边界意识值得说清楚：

`mount` 插件只读 `/proc/mounts`，不会去 `stat` 挂载路径。

这意味着：

- 它不会因为 NFS 卡死把自己也卡住
- 但它检查的是“有没有挂载上”，不是“挂载点是否真的可用”

这就是成熟插件设计该有的克制。

NFS 可用性该交给 `disk` 的 writable 检查或 `ping` / `net` 去补，不应该混在一起。

### 3. 安全模块基线：`secmod`

`secmod` 是我觉得很有代表性的一个插件，因为它非常贴近真实运维习惯。

最典型的故事就是：

> 某位同学为了排障，临时把 SELinux 调成 permissive。问题解决后，人走了，状态没改回来。

然后接下来的几个月，系统一直在一个“看起来没坏，但已经不符合安全期望”的状态里运行。

这就是典型的漂移。

catpaw 在这块的设计也很对：

- 插件本身不带立场
- 它不替你决定 SELinux 必须 enforcing 还是 disabled
- 它只问一个问题：当前状态是不是和你的期望一致

配置也非常直接：

```toml
[[instances]]
interval = "60s"

[instances.enforce_mode]
expect = "enforcing"
severity = "Warning"

[instances.apparmor_enabled]
expect = "yes"
severity = "Warning"
```

这类检查的价值在于：

它不是为了让 dashboard 更好看，而是为了避免“临时操作永久化”。

### 4. 时间同步基线：`ntp`

很多团队只有在证书校验、日志排序、token 认证开始异常时，才意识到时间不对。

但其实 NTP 这一层非常适合被提前盯住：

- 是否同步
- 偏移量是否过大
- stratum 是否异常

catpaw 的 `ntp` 插件也比较实用，默认会自动探测 chrony、ntpd、timedatectl：

```toml
[[instances]]

[instances.sync]
severity = "Critical"

[instances.offset]
warn_ge = "100ms"
critical_ge = "500ms"

[instances.stratum]
warn_ge = 10
critical_ge = 16
```

这类检查很适合作为平台基线的一部分长期存在。

因为时钟问题一旦放大，影响通常是跨系统、跨层次的。

## 只靠定时检查还不够，变更后还要主动巡检

讲到这里，还差最后一层。

因为哪怕你已经配好了基线检查，仍然会遇到一种非常现实的场景：

- 刚做完内核调优
- 刚改完挂载配置
- 刚完成一次主机加固
- 刚做完 Redis / systemd / 网络相关变更

这时候你不想等下一次事故来验证。

你更想立刻问一句：

> 现在这台机器整体看起来正常吗？

这就是 `catpaw inspect` 最适合出现的地方。

它不是替代周期检查，而是把“主动确认”这个动作也标准化了。

比如：

```bash
./catpaw inspect sysctl
./catpaw inspect disk /data
```

它更像是在做一件以前大家常靠经验和临时命令完成的事：

- 我刚改了配置，帮我检查一下这一层还有没有明显问题
- 我想在变更窗口里快速做一轮体检
- 我不想等告警触发才第一次发现异常

这类“主动巡检”能力，对成熟团队特别重要。

因为成熟 SRE 的工作，本来就不只是接告警。

还包括在问题演化成告警之前，提前把风险看出来。

## 一套更成熟的值班习惯，应该是什么样

如果把这些能力合起来，我更推荐的工作方式其实很朴素：

### 平时

- 用 `sysctl`、`mount`、`secmod`、`ntp` 守住长期基线
- 用 `disk`、`cpu`、`mem`、`net`、`systemd` 盯显性异常

### 变更后

- 跑一轮 `inspect`
- 对关键层做主动确认

### 真出问题时

- 让告警触发自动诊断
- 或直接进 `catpaw chat` 做交互排查

这 3 层合起来，才比较像完整的机器侧治理闭环：

```text
平时守基线 -> 变更后做巡检 -> 出问题再诊断
```

而不是：

```text
什么都不做 -> 等告警响 -> SSH 上机现查
```

## 如果你现在就想开始，最值得先配的就是这 4 项

如果你想最低成本地把“预防式治理”补起来，我建议优先顺序就是：

### 1. `sysctl`

先把最关键的几个内核参数写成机器基线。

### 2. `mount`

先把数据盘、备份盘、关键安全挂载选项写清楚。

### 3. `secmod`

把 SELinux / AppArmor 的期望状态固定下来。

### 4. `ntp`

至少把同步状态和偏移量看起来。

这 4 个加起来，并不会让你的监控体系显得“更豪华”。

但它会让你的系统更不容易在无人注意的地方悄悄变坏。

## 最后总结

真正成熟的 SRE，不是等告警响了才开始看机器。

更成熟的状态是：

- 你知道机器应该长什么样
- 你把这种“应该”写成了可执行的基线
- 你在变更后会主动确认
- 你把漂移当成故障的前兆，而不是事故后的复盘素材

这也是我觉得 catpaw 值得继续往下写的原因。

它不只是让你在半夜少敲几条命令。

它更大的价值，是把很多原本只存在于资深 SRE 脑子里的“运维常识”，变成一组可以持续执行的检查和巡检动作。
