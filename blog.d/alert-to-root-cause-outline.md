# 文章提纲：告警发出来之后，谁来查根因？

## 文章定位

- 主标题：告警发出来之后，谁来查根因？
- 备选标题：
  - 从"磁盘 95%"到"到底是谁写满了盘"：告警闭环该怎么做
  - 传统监控只会报警，catpaw 开始帮你定位问题
  - 告警不是终点，根因才是 SRE 真正想要的东西
- 目标读者：On-call SRE、值班开发、平台工程师
- 文章目标：让读者理解 catpaw 不是只会发告警，而是在做"异常发现 + 根因初筛"的闭环

## 核心观点

一条只写着"disk usage 97%"的告警，只完成了故障处理的前 20%。真正有价值的系统，应该在告警触发后顺手完成第一轮诊断，把"是什么坏了"推进到"大概率为什么坏了"。

## 建议结构

### 1. 开头场景：凌晨 3 点收到一条没有上下文的告警

- 用磁盘、OOM 或 conntrack 其中一个案例开篇
- 先让读者代入传统流程：
  - 收到告警
  - 登录机器
  - 回忆命令
  - 一层层追查
- 点出问题：多数监控系统在"发现异常"这一步就停了

### 2. 为什么多数告警系统只解决了 20% 的问题

- 告警告诉你"现象"
- 排障要回答的是：
  - 根因是什么
  - 影响范围有多大
  - 下一步该查什么
- 可用小表格对比：
  - 传统告警
  - 带根因初筛的告警

### 3. 一个真正可用的告警闭环，需要哪些环节

- 异常检测
- 标准化事件
- 去重 / 聚合 / 降噪
- 根据告警上下文自动选择诊断动作
- 输出结构化报告
- 报告和告警一起进入值班链路

这里可以自然引出 catpaw 的架构，但重点放在问题拆解，不要先讲产品。

### 4. catpaw 是怎么把这条链路补上的

- 用一张简化链路图说明：
  - plugin check
  - event engine
  - notifier
  - diagnose aggregator
  - AI diagnose engine
  - report forward
- 可以结合 `docs/dev-guide.md` 写几个关键细节：
  - 同一目标的多个告警可在窗口内聚合
  - AI 不是胡猜，而是调用 70+ 诊断工具
  - 诊断报告会作为新事件再次推送

### 5. 具体案例：以"磁盘空间告急"或"conntrack 溢出"为例

- 告警原文是什么
- 如果靠人工，要查哪些命令
- catpaw 可能会怎么查：
  - 磁盘：`df`、inode、目录占用、日志、core dump
  - conntrack：当前使用率、上限、丢包证据、相关内核参数
- 最终报告应该长什么样

这部分最好直接给一段示例 Markdown 诊断报告。

### 6. 工程上为什么这件事以前很难做

- 告警太多，不能每条都深查
- 上下文不够，AI 容易胡说
- 诊断动作要安全，不能默认执行危险命令
- 需要冷却、超时、并发控制，避免把故障机器查死

这部分适合建立技术可信度，读者会感觉你不是在卖概念。

### 7. 这套能力适合放在哪些场景

- 值班告警
- 重大异常的第一轮初筛
- 跨团队交接时提供统一上下文
- 经验不足的开发值班场景

### 8. 5 分钟上手

- 开启一个基础插件
- 打开 console notify
- 配置 AI 模型
- 演示一次低阈值触发
- 看一条告警和一条诊断报告分别长什么样

### 9. 收尾

- 收束到一句话：
  - 传统告警回答"坏了没有"
  - 告警闭环回答"为什么坏，接下来先查什么"
- CTA：
  - 让读者先在测试机开一个插件试试
  - 或引导到 `README.md` / release 页面

## 写作注意点

- 不要把重点写成"AI 很聪明"
- 重点应放在：告警闭环如何减少人肉排查成本
- 读者最关心的是：
  - 这事为什么值得做
  - 会不会误查
  - 会不会太重
  - 怎么接进我现有值班链路
