# catpaw 内容路线图（SRE / DEV）

## 目标

- 主要目标：扩大 catpaw 在 SRE / DEV 人群中的认知
- 次要目标：建立"不是 AI 玩具，而是能落地排障"的信任感
- 内容原则：每篇文章都要让读者即使不装 catpaw，也能学到一套判断方法、监控点或排障流程

## 当前已覆盖

- 项目全景介绍
- Linux 内核层暗坑监控
- `catpaw chat` 的自然语言排障

## 当前最值得补的空白

1. 告警之后到根因之前，这段链路应该怎么设计
2. catpaw 与 Prometheus / Grafana / Alertmanager 的边界和配合方式
3. 垂直场景专题，优先 Redis
4. 预防式巡检、基线与配置漂移
5. 外部上下文接入后，AI 排障能力为什么会明显提升

## 推荐内容支柱

### 支柱 1：监控盲区

聚焦"为什么大盘全绿，服务还是坏了"。这类文章最容易打中资深 SRE。

### 支柱 2：告警闭环

聚焦"发现异常之后，系统能否顺手完成一轮根因初筛"。这是 catpaw 最强差异点。

### 支柱 3：场景专题

按 Redis、systemd、证书、容器、K8s 网络等专题展开，帮助读者把 catpaw 映射到自己的环境。

### 支柱 4：工程化落地

聚焦部署、接入现有监控栈、降噪、冷却、聚合、通知渠道、巡检策略。

## 先写哪两篇

### 1. 告警发出来之后，谁来查根因？

- 推荐理由：这是 catpaw 最核心的差异点，且能直接击中 SRE 的值班痛点
- 文章目标：让读者理解"告警闭环"的价值，不再把 catpaw 只看成又一个 Agent
- 证据来源：
  - `README.md` 的架构图、AI 诊断能力、通知链路
  - `docs/dev-guide.md` 中的聚合器、诊断引擎、事件流
  - `blog.d/catpaw-intro.md` 中的场景化表达

### 2. Prometheus 很强，但它不负责半夜帮你查机器

- 推荐理由：这篇能提前回答最常见的定位问题，减少读者误解
- 文章目标：清楚说明 catpaw 补的是"异常检测 + 初步根因分析"而不是指标采集
- 证据来源：
  - `README.md` 中"not a metrics collector"的产品定位
  - `blog.d/linux-silent-killers.md` 中的多个盲区案例
  - `docs/dev-guide.md` 中 catpaw 的事件模型和诊断链路

## 写作统一模板

1. 用真实故障或值班场景开头
2. 说明传统监控 / 传统排障为什么慢
3. 提炼该问题真正要看的监控点
4. 介绍 catpaw 在这一步补了什么
5. 给出配置、命令、告警或诊断示例
6. 最后总结一份读者能带走的 checklist
